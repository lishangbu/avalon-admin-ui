# CRUD 页面工作流

## 适用场景

- 新增管理页
- 现有列表页需要补查询、弹窗表单或删除流程
- 常规 CRUD 页面需要对齐仓库里的既有模式

如果是全新的标准 CRUD 页面，优先先跑 `scripts/scaffold_crud_page.py`，再对生成结果做定制，而不是从空白文件开始手写。

## 默认文件形状

- `index.tsx`：页面主体、查询表单、表格、弹窗、Query hook
- `service.ts`：列表、详情、新增、更新、删除接口
- `types.ts`：查询类型、表单类型、本地 view 类型

不是每个页面都必须有三者，但这是默认起步形状。

## 实现步骤

### 1. 先建好页面状态模型

- 查询表单值和后端查询类型分开定义，方便做 trim 和字段映射
- 分页状态使用 `page`、`pageSize`
- 弹窗开关、编辑中的 id、保存中和详情加载状态留在本地组件状态里

### 2. 先搭服务层

- 每个接口都用清晰命名的异步函数包装
- 不直接在页面里用原始 axios
- GET 查询参数优先走 `compactParams` 或通用分页辅助函数
- 后端返回结构如果不顺手，先在服务层或 normalizer 里整理干净

### 3. 用 TanStack Query 接列表和选项数据

- `queryKey` 必须包含所有会影响服务端结果的条件
- 用 `enabled` 处理权限判断或依赖未就绪场景
- 分页表格可以继续用 `keepPreviousData`
- 手动刷新时，优先使用 `ensureQueryData()`、`refetch()` 或统一的 `loadRows()` 封装

### 4. 组装页面外壳

- 外层优先使用 `PageContainer`
- 支持筛选的页面，查询表单通常放在表格上方
- 没有页面级查询权限时，渲染 `Result status="403"`
- 页面级按钮一般放在 `extra` 区域，行内操作放在操作列

### 5. 处理增删改

- 新增和更新先做表单校验，再根据是否有 id 决定调用哪个接口
- 删除优先用 `Popconfirm`
- 成功后在页面层给出提示，并刷新列表
- 弹窗关闭或提交成功后，重置相关表单和临时状态

## 仓库内的常见约定

- 操作按钮优先用 `PermissionGuard`
- 表单转请求体时，优先用页面内的小函数集中处理
- 类似启用状态这种枚举值，优先用 `Tag` 之类的轻量视觉反馈
- `rowKey` 要稳定，不要依赖不可靠字段
- 弹窗表单字段多时，优先用响应式网格布局

## 动手前检查

- 先补页面级查询权限，再接按钮权限
- 每个服务端筛选条件都要进入 `queryKey`
- 可选字符串字段发给后端前先做 trim
- 刷新逻辑尽量显式，不要把刷新寄希望于隐式缓存行为
- 如果列表接口返回字段不够编辑页使用，编辑时单独拉详情
- 如果后端 id 是数字，优先在 normalizer 中转成字符串
- 脚手架生成的 `keyword`、`code`、`name`、`remark` 等默认字段，如果和真实业务不符，要尽快换掉

## 避免的做法

- 在渲染逻辑里直接拼请求体
- 常规读取流程绕开 TanStack Query
- 在多个组件重复写权限字符串
- 每个页面各自造一套请求客户端
